## 1. Executive Summary

SignalStack SMS is a **highly engineered, demo-safe, AI-agent-built backend platform** for a multi-tenant US A2P 10DLC SMB SMS/MMS SaaS (Twilio-first, compliance-first, queue + workers, fake AI, usage metering, readiness audits, credential rotation, hard external-impact gates).

**What exists and is strong:**
- World-class local/CI validation harness (`npm run validate` runs 15+ gated steps with forced `DEMO_MODE=true`, live flags off).
- Substantial production-grade backend: Prisma (13 migrations), tenant-scoped repositories (contacts, campaigns, inbox, compliance, provider, readiness, etc.), Zod validation at every boundary, CSV import, send preflight + compliance gates, dummy + Twilio provider skeletons, BullMQ-optional queue, workers, Twilio webhook handlers (inbound/status with idempotency/signature), fake AI adapter, local usage/billing records.
- ~30+ API routes covering the CANONICAL map (orgs, contacts/imports, campaigns+preflight/schedule/cancel, inbox full actions, ai/*, analytics, billing, settings/*, webhooks, demo helpers).
- Elaborate **meta "Local Launch Dashboard + 35 settings/operations pages"** (`/`, `/settings/*`, `/demo`) that are read-only/admin inventories, projections, vocabularies, and audit surfaces with runtime freezes, shape assertions, and cross-linked E2E coverage. This layer is sophisticated self-documentation for agentic development.
- Strong contracts, docs, demo seed (realistic opted-in/out contacts, campaigns, conversations), e2e smoke + meta demo-path tests.

**What is missing / divergent (the core problem):**
- **Zero customer-facing product UI.** No pages for the actual SMB user workflows: contact management table + import wizard, campaign composer (with AI generate, variable preview, recipient list, preflight results), shared inbox (thread view, reply, assignment, notes, resolve), analytics dashboard, templates editor, go-live interactive checklist. The only UI is the internal operator/audit console.
- The "investor demo goal" in CANONICAL §1 (10-minute end-to-end: CSV import → compliance → AI copy → schedule → simulated replies/STOP/HELP → AI summary/lead qual → analytics → go-live) **cannot be performed by a non-dev in the browser today**.
- Schema in `prisma/schema.prisma` is simpler/renamed vs the rich sketch in CANONICAL (Organization/AppUser/MembershipRole vs Org/User, minimal Message/Compliance/AI models, no AiRun/LeadQualification tables, etc.). Code works against what exists (some data in JSON or partial).
- `PLAN.md` is severely outdated: claims "Milestones 0–10 completed + 300+ Post-MVP items" (many repetitive "read-only local X operations view / shared operator projection / vocabulary hardening / detached array count coverage"). These were generated by prior runs; actual product value is not shipped.
- The project has meta-evolved: the "product" became the **engineering harness, contract system, ops surfaces, and agent coordination docs** themselves. Previous 25 codex-runs produced enormous self-referential artifacts (`codex-runs/`, `lib/operations/*` with 100s of lines of freeze/assert/projection code, 35 settings pages each doing "render existing X only; does not mutate/call live").
- README and many docs now describe the meta surfaces as the demo story ("renders existing... only; does not send SMS...").

**Strategic fork:** The excellent engine + gates + contracts + meta tooling are a **fantastic foundation**. The missing piece is the thin product UI layer that lets a real user experience the SMS value prop on top of it.

**Recommendation (Ultra Plan core):**  
**Re-anchor to the original product promise.** Treat the current meta-ops console as valuable "Advanced / Operator / Debug / Agent surfaces" (keep them, perhaps move under `/ops` or "Advanced" nav). Build a focused, minimal product UI for the 6–8 flows needed for a credible investor demo. Update all source-of-truth docs to describe reality + the new direction. Clean the bloat in PLAN.md. Deliver a runnable "10-minute demo path" that actually exercises the *product*, not just the scaffolding.

This is achievable quickly because the hard parts (data model, gates, preflight, queue, compliance, AI, APIs, seed, validation) are already done and gated.

## 2. Context & History (Why We Are Here)

- Started as "SMS version of the CRM repo pattern" (source-of-truth docs, contracts-first, agent ownership zones, worktrees, one-command gate, demo-safe defaults, hard gates for live SMS/billing/secrets).
- CANONICAL_IMPLEMENTATION_PLAN.md is the bible: ambitious full product + investor demo + detailed architecture/contracts/milestones/agent work orders/CI/worktree/merge/rollback rules.
- Agents (especially codex) followed the "maximum automation + read everything + plan" loop 25+ times, producing:
  - 25 FULL_PLAN_RUN_*.md + SUMMARIES (massive context rehydration each time).
  - Granular Post-MVP checklists (the 300+ items in current PLAN.md).
  - The entire `lib/operations/` + 35 `app/settings/*` pages + operator-surfaces.ts with frozen projections, route-shape validators, vocabulary hardening, no-impact summaries, detached counts, etc.
  - E2E that asserts the meta inventory rather than product flows.
- Result: exceptional engineering rigor and self-audit capability, but the "SMB texting app" UI and happy-path product demo were never built. The meta work satisfied the checklist volume but not the original MVP promise.
- Current package.json still says "Milestone 0 scaffold"; reality is "advanced meta-engine + partial backend + zero product UI".
- AGENTS.md + LOCAL_GATE.md correctly enforce "start with M0, no product features until validation/docs/contracts/CI/demo-safe defaults" — but those are now satisfied at the *engine* level.

The codebase is not "broken" — it is **over-optimized for the meta-problem** ("how do we let AI agents safely build a safe SMS SaaS") and under-optimized for the user-problem ("let an SMB owner import 75 contacts and text them safely").

## 4. Root Cause Analysis

1. **Checklist > Outcome.** The Post-MVP list in PLAN.md + repeated codex runs created a target of "cover every possible operations surface / vocabulary / projection invariant." Agents delivered exactly that (impressive engineering) instead of "make a user able to send a text message from the UI."

2. **Self-reinforcing meta loop.** Each planning run re-read the entire (growing) corpus, added more surfaces to satisfy the previous plan's "hardening" items, grew the ops lib and settings/, updated E2E to cover the new surfaces, without a forcing function to build the product UI.

3. **No "product UI agent zone" ownership.** Worktree rules and agent missions focused on backend/contracts/docs/meta; frontend-ui zone was never strongly exercised for the core flows.

4. **Demo path became meta path.** The "investor demo" was redefined as "see all the safe engineering surfaces" rather than "use the texting product."

5. **Schema followed implementation, not the sketch.** Practical, but docs fell out of sync.

The good news: all the *hard and risky* parts (compliance, tenant isolation, external gates, queue durability, webhook idempotency, contract checking, validation) are solved. Adding the UI is "just" wiring existing APIs into nice forms/tables + a bit of client state + navigation shell.

## 5. Recommended Strategic Direction

**Primary: "Product-First Recovery" (Option A)**

- Keep 100% of the safety, contracts, backend, ops surfaces, and validation.
- Add a **product UI layer** (new `app/(product)/` or `app/(dashboard)/` parallel to settings, or integrate cleanly into the existing shell with two modes: "Product" vs "Operator").
- Make the root `/` or a prominent "Launch Demo" button start the 10-min product story.
- Move or nest the heavy ops surfaces under `/ops` or "Advanced → Operator Surfaces" so they don't dominate the experience for a product demo.
- Update CANONICAL/PLAN/ARCHITECTURE/README to describe the dual nature honestly and prioritize the product flows.
- This delivers real value and makes the impressive engineering *visible through the product* (e.g. preflight blocks show compliance reasons in the UI, STOP updates inbox + contact instantly, AI suggestions appear in composer, etc.).

**Alternative: "Meta Showcase" (Option B)** — Accept that the artifact is the safe backend + ops console for AI-built SaaS. Thin the SMS product UI to a few demo buttons that call the existing APIs. Position the repo as a reference implementation / starter for "how to build a gated, auditable, agent-maintainable SaaS with local operator visibility." Less market appeal for the SMS use case.

**Decision:** Option A is strongly recommended. The original market, buyer, and demo promise were the texting product. The engine exists to power it. The meta work is reusable scaffolding.

### Phase 1: Product UI Foundations — Investor Demo Path (core value, 1–2 weeks)

**Goal:** A non-technical user can sit at the keyboard and complete the 10-point CANONICAL investor demo in the browser using real UI (not curl or settings pages). The existing backend does 95% of the work; we just need forms, tables, and navigation.

**Scope (minimal, demo-grade, not pixel-perfect):**
- New route group or section: `app/(product)/` or keep under root with clear "Product" vs "Operator" nav in the shell.
- Layout/shell update: top-nav or side-nav with "Product Demo" | "Operator Surfaces" | "Go Live Checklist".
- 6–8 pages (RSC + minimal client where needed):
  1. **Dashboard** (`/dashboard` or `/` product mode): analytics overview cards (reuse /api/analytics/overview), recent conversations, quick actions (import, new campaign, inbox).
  2. **Contacts** (`/contacts`): table (paginated/filter by consent/status/tag), search, "Import CSV" button → modal or page using existing `/api/contacts/imports` + csv lib (show preview, errors, success → list updates). Show consent badges.
  3. **Campaigns** (`/campaigns`): list (status, scheduled, recipient counts), "New Campaign" → composer:
     - Name, body (textarea with variable picker {{firstName}} etc.), media URLs.
     - AI "Generate copy" button (calls /api/ai/campaign-copy with context, inserts).
     - Recipient selector (lists, tags, segments, or all opted-in).
     - "Run Preflight" (calls /api/campaigns/[id]/preflight, shows blocked reasons with compliance details — this is where the gates shine).
     - "Schedule" (date/time or now, calls schedule, shows queue job).
  4. **Inbox** (`/inbox`): list of conversations (status, contact, last message, assigned), click → thread view:
     - Messages (out/in, status).
     - Reply composer (calls inbox messages API, handles STOP/HELP side effects via backend).
     - Demo "Simulate Inbound" buttons (STOP, HELP, lead, normal — reuse /api/demo/inbound).
     - Assign, add note, resolve (reuse existing APIs).
  5. **Templates** (simple list + editor, reuse /api/templates + render).
  6. **Analytics** (reuse overview + charts — minimal Recharts or native).
  7. **Compliance / Settings** (link to existing strong /settings/compliance, /numbers, /provider — they already have forms for profile, credentials metadata, rotations, readiness audit).
  8. **Go-Live / Readiness** (reuse/extend /settings/readiness-audit + go-live checklist that queries real gate state from compliance + provider + org flags).

**Technical approach (respect existing):**
- Server Components for lists/details (call repos or API routes internally).
- Server Actions or fetch to the existing APIs for mutations (import, preflight, schedule, reply, etc.).
- Use the strong Zod schemas for client validation too.
- Add minimal UI primitives (cards, tables, modals, badges, toasts) — either hand-rolled Tailwind or install shadcn/ui (components.json already present) for speed/quality.
- Polling (every 5s) for queue status / new inbound (as planned in CANONICAL).
- Theme consistent with existing ops pages (slate/teal).
- Mobile-friendly but desktop-first for demo.

**Demo Script (new or updated E2E):**
- Reset demo org (button or seed).
- Import the demo CSV.
- Open a blocked contact → see why (consent).
- Create campaign, generate AI copy, pick 10 opted-in, preflight (show 0 blocked or the expected), schedule.
- Go to inbox, simulate 3 inbounds (one STOP — observe contact status change + future preflight blocks it, one lead), AI-summarize one, qualify, reply, assign, resolve.
- Refresh analytics — numbers moved.
- View compliance checklist + provider readiness.
- "This is what an SMB user experiences; all the gates, audits, and worker processing happened safely."
